調査の読み方How to Read This Investigation

なぜこの調査が難しく、なぜ面白いのか。データの羅列を読む前に、この調査が何を問い、何が見えて、何が見えないのかを理解してほしい。

Why this investigation is hard, and why it is interesting. Before reading the data, understand what this investigation is asking, what can be seen, and what cannot.

C14 portability pilot gfx1201 / RDNA4 Phase A–F

C14 portability pilot とは何かWhat is the C14 Portability Pilot?

この調査は、MI25（GCN5/Vega）で確立した調査方法論を RDNA4（RX9070XT）に移植する試みです。「移植」という言葉は単純に聞こえますが、実態は方法論そのものの検証実験です。

This investigation is an attempt to transplant the methodology established for MI25 (GCN5/Vega) onto RDNA4 (RX9070XT). The word "transplant" sounds simple, but in practice it is a validation experiment for the methodology itself.

MI25 調査では、rocprofv3 による kernel 追跡・ROCBLAS_LAYER ログ・プロセスマップ観測の3本柱で「何が実行されたか」を直接確認できました。RDNA4 では、この3本柱が全て機能しません。そこで問うべき問いが変わります：

In the MI25 investigation, three pillars — rocprofv3 kernel tracing, ROCBLAS_LAYER logging, and process map observation — allowed direct confirmation of "what was executed." In RDNA4, all three fail. This shifts the central question:

「MI25 でできたことを RDNA4 でもできるか」ではなく、「MI25 で当然だった観測窓が使えない状況で、何が観測可能で何が観測不可能かを切り分けられるか」。これが C14 pilot が問うていることです。

"Can we do in RDNA4 what we did in MI25?" — No. The question is: "In a situation where the observation windows that were standard for MI25 are unavailable, can we distinguish what is observable from what is not?" That is what C14 asks.

答えが「できない」であっても、「何がなぜ観測できないのか」を特定できれば、それは調査の失敗ではなく調査の成果です。

Even if the answer is "no," identifying what cannot be observed and why constitutes a research finding — not a failure.

Observer 問題：「観測しようとしたら壊れた」は発見であるThe Observer Problem: "It Broke When We Tried to Observe" Is a Finding

RDNA4 調査で最初に直面する壁は、観測ツールが軒並み機能しないことです。これは「調査が失敗した」のではなく、「RDNA4 の runtime model がツールチェーンの想定と異なる」という発見です。

The first obstacle in the RDNA4 investigation is that observation tools fail across the board. This is not "the investigation failed" — it is the discovery that RDNA4's runtime model differs from what the toolchain assumes.

重要なのは、ツールが壊れた理由を特定できることです。rocprofv3 wrap が CPU fallback を引き起こすのは、Ollama runner の GPU bootstrap discovery が LD_LIBRARY_PATH の書き換えに依存しており、profiler の介在がその前提を崩すからだと読めます。ptrace_scope=1 は OS のカーネルセキュリティ設定であり、これを回避しようとすることはスコープ外です。これらは「観測の失敗」ではなく「観測窓の構造的限界」です。

What matters is being able to identify why the tools fail. rocprofv3 wrap causing CPU fallback can be read as: the Ollama runner's GPU bootstrap discovery depends on LD_LIBRARY_PATH rewriting, and profiler interposition breaks that assumption. ptrace_scope=1 is an OS kernel security setting — circumventing it is out of scope. These are "structural limits of the observation window," not "observation failures."

「見えない」の切り分け：観測窓の問題か、target 固有の特徴かDistinguishing "Invisible": Observation Window Limit vs. Target-Specific Characteristic

何かが観測されないとき、それには2種類の原因があります。

When something is not observed, there are two possible causes.

観測窓の問題：ツールが壊れているため、起きていることが見えない。例：rocprofv3 が CPU fallback を引き起こすため、GPU 上での実行が追跡できない。

Observation window limit: The tool is broken, so what is happening cannot be seen. Example: rocprofv3 causes CPU fallback, so GPU-side execution cannot be traced.

target 固有の特徴：実際に起きていない、または起きる経路に入っていない。例：MIOpen が /proc/maps に現れないのは、Ollama の LLM 推論経路が MIOpen を使用しないからだと読める（bundle にも含まれていない）。

Target-specific characteristic: It is genuinely not happening, or the path leading to it is not entered. Example: MIOpen not appearing in /proc/maps can be read as the Ollama LLM inference path not using MIOpen (it is also absent from the bundle).

この切り分けが、本調査の中心的な知的作業です。dispatch-safe observer がない状態では、多くの「見えない」は前者（観測窓の問題）と後者（実際に起きていない）が区別できません。それを区別できた事例・できなかった事例を正直に記録することが、この調査の価値になります。

This distinction is the central intellectual work of this investigation. Without a dispatch-safe observer, many "invisibles" cannot be disambiguated as the former (observation window limit) vs. the latter (genuinely not happening). Honestly recording which cases could be distinguished and which could not is where this investigation's value lies.

load ≠ dispatch：ライブラリが読み込まれることと演算が実行されることは別の問題load ≠ dispatch: Library Loading and Operation Dispatch Are Separate Questions

Phase C の調査で、Ollama runner の /proc/maps に libhipblaslt.so.0.10.60303 が記録されていることが確認されました。これは「hipBLASLt がロードされている」という事実です。しかしここから「hipBLASLt の演算が LLM 推論ループで dispatch された」とは言えません。

Phase C confirmed that libhipblaslt.so.0.10.60303 appears in the Ollama runner's /proc/maps. This is the fact: "hipBLASLt is loaded." From this, however, it cannot be stated that "hipBLASLt operations were dispatched during the LLM inference loop."

なぜ区別が必要か。deepseek-r1-distill-qwen-7b:q4_k_m の Q4_K_M 推論では、ne11（バッチサイズ相当）の値に応じてカーネルが分岐します。decode フェーズ（トークン生成）の多くは ne11 が小さく、MMVQ（mul_mat_vec_q）か MMQ（mul_mat_q）の custom HIP kernel が主経路として使われると推測されます。BLAS fallback（hipBLAS → rocBLAS → hipBLASLt）に到達するのは ne11 が大きい prefill フェーズか、一部の特定な shape の場合に限られる可能性があります。

Why does the distinction matter? In Q4_K_M inference with deepseek-r1-distill-qwen-7b:q4_k_m, kernels branch based on ne11 (analogous to batch size). Most of the decode phase (token generation) likely has small ne11, where MMVQ (mul_mat_vec_q) or MMQ (mul_mat_q) custom HIP kernels serve as the primary path. The BLAS fallback (hipBLAS → rocBLAS → hipBLASLt) may only be reached during the prefill phase with large ne11, or for specific shapes.

dispatch-safe observer がない現状では、「ロードされている」と「実行された」の間の溝を埋めることができません。これは本調査の現時点での根本的な限界です。

Without a dispatch-safe observer, the gap between "is loaded" and "was executed" cannot be bridged. This is the fundamental current limitation of this investigation.

phase proxy：runtime を壊さずに dispatch 閾値を推測するPhase Proxy: Inferring Dispatch Thresholds Without Breaking the Runtime

dispatch-safe observer が存在しない中で、「どのカーネル経路が使われたか」を間接的に推測する手法として phase proxy を採用しました。

With no dispatch-safe observer available, phase proxy was adopted as an indirect method to infer which kernel path was taken.

Ollama の API レスポンス JSON には prompt_eval_count（prefill トークン数）と eval_count（生成トークン数）・eval_duration（生成時間）が含まれます。prompt_eval_count は ne11 の代理指標として機能します。

Ollama's API response JSON includes prompt_eval_count (prefill token count), eval_count (generated tokens), and eval_duration (generation time). prompt_eval_count serves as a proxy indicator for ne11.

proxy の限界：prompt_eval_count はトークン数であり、カーネル launch 時点での実際の ne11 テンソル次元を直接読んでいるわけではありません。chunked prefill や KV cache の挙動によって、同じトークン数でも dispatch される ne11 の値が変わる可能性があります。「MMVQ 対象」「MMQ 対象」という分類は推測であり、dispatch-safe observer による直接確認が本来必要です。

Proxy limitation: prompt_eval_count is a token count, not a direct reading of the actual ne11 tensor dimension at kernel launch. Chunked prefill or KV cache behavior could cause the dispatched ne11 value to differ even for the same token count. Classifications like "MMVQ range" or "MMQ range" are inferences — direct confirmation by a dispatch-safe observer is the proper requirement.
