RX9070XT Path Observability

RX9070XT の経路観測記録です。plain runtime-path alive で止まらず、source tracing / bundle isolate / disassembly / runtime correlation を段階的に固定します。

Path observation log for RX9070XT. This page goes beyond plain runtime proof-of-life and fixes source tracing, bundle isolation, disassembly, and runtime correlation step by step.

観測済み事実Observed Facts

  • fresh runtime は bundle-first で /usr/local/lib/ollama/rocm を使い、libggml-hip / hipBLAS / rocBLAS / amdhip64 を実際にロードします。
  • libggml-hip.so には RDNA4 / gfx1201 向けの custom hsaco 群が埋め込まれており、BLAS 側には rocblas/library/Kernels.so-000-gfx1201.hsaco の Cijk_* family が共存します。
  • custom 側では Q4_K dequant、mul_mat_vec_q、exact mul_mat_q、MMQ repack、Flash Attention、RoPE の family を bundle 単位で isolate 済みです。
  • The fresh runtime is bundle-first, using /usr/local/lib/ollama/rocm and actually loading libggml-hip, hipBLAS, rocBLAS, and amdhip64.
  • libggml-hip.so embeds RDNA4/gfx1201 custom hsaco families, while the BLAS side also ships Cijk_* kernels in rocblas/library/Kernels.so-000-gfx1201.hsaco.
  • On the custom side, Q4_K dequant, mul_mat_vec_q, exact mul_mat_q, MMQ repack, Flash Attention, and RoPE families have already been isolated bundle-by-bundle.

現時点での知見Current Findings

  • short resident q4_k_m case は pure-MMVQ ではなく、decode 側は MMVQ-compatible、prompt 側は 11-12 count window で MMQ-eligible とみなすのが現時点の最良整理です。
  • larger quantized path 用の exact Q4_K mul_mat_q object は bundle_0096 として isolate 済みです。
  • prompt_eval_count = 290 の long prompt probe は、MMQ ceiling 超過か internal chunking かを切り分けるべき境界 probe として扱っています。
  • The short resident q4_k_m case should no longer be treated as pure MMVQ: decode-side remains MMVQ-compatible, while the prompt-side 11-12 count window is already MMQ-eligible.
  • The exact Q4_K mul_mat_q object for the larger quantized path is now fixed as bundle_0096.
  • The long-prompt probe at prompt_eval_count = 290 is treated as a boundary probe for either MMQ ceiling crossing or internal chunking.

未確定事項Open Questions

  • short live case が bundle_0096 の MMQ family をどの程度踏んだかは、まだ直接は確定していません。
  • BLAS-side Cijk_* family が live short case に入ったかも未確定です。
  • prompt_eval_count は有用な proxy ですが、kernel launch 時の正確な ne11 を直接読んでいるわけではありません。
  • We still do not know how strongly the short live case actually exercised the bundle_0096 MMQ family.
  • Whether the BLAS-side Cijk_* family entered the short live case is also unresolved.
  • prompt_eval_count is a useful proxy, but it is not a direct readout of the exact ne11 seen at kernel launch.

追記予定Planned

  • bundle_0096 の live phase ownership を、非破壊 observer と phase proxy の範囲でさらに詰めます。
  • long prompt boundary が fallback か chunking かを切り分けます。
  • We will narrow the live phase ownership of bundle_0096 further using non-distorting observers and phase proxies.
  • We will separate whether the long-prompt boundary is fallback-driven or chunking-driven.
