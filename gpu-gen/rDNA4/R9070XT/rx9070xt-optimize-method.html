One question drives this page: is there any means to directly read which GPU kernel is active on Ollama running on RX9070XT?
The answer is currently No. Wrap, attach, and layer logging all fail. What remains by elimination is only the phase proxy (response JSON).
This is also the core reason why the C14 portability pilot proceeds via multi-layer inference rather than direct observation.

驚きポイント： 通常の profiling 手段（wrap・attach・layer log）を使おうとすると、GPU 自体が動かなくなるか、OS に止められるか、見たいものが見えないかのいずれかになる。 つまり 「何が動いているか」は、標準的な手段では読めない。 The surprising part: Every standard profiling approach (wrap, attach, layer log) either causes the GPU to stop being used, is blocked by the OS, or lacks sufficient granularity to see what matters. In short, there is no standard means to read "what is running."

このページの結論 Page Conclusion

試した直接 observer はすべて機能しない（wrap → CPU fallback、--attach → ptrace ブロック、ROCBLAS_LAYER → 粒度不足）。
消去法の末に残るのは phase proxy（response JSON の prompt_eval_count）だけ。 これは「優れた手段を選んだ」ではなく、「他がすべて機能しない状況で唯一生き残った手段」。
この事実が、C14 pilot が直接観測ではなく多層証拠の積み上げで進む理由を直接説明する。 All direct observer approaches tried are non-functional (wrap → CPU fallback, --attach → ptrace blocked, ROCBLAS_LAYER → insufficient granularity).
By elimination, only the phase proxy (prompt_eval_count from response JSON) remains. This was not chosen for its quality — it is the last approach standing after all others failed.
This is the direct explanation for why the C14 pilot proceeds via multi-layer inference rather than direct observation.

手段① — rocprofv3 wrap（結果：CPU fallback） Approach 1 — rocprofv3 wrap (Result: CPU fallback)

観測ポイント： rocprofv3 で ollama serve をラップすれば、per-kernel dispatch を記録できるか？

Observation target: Can wrapping ollama serve with rocprofv3 record per-kernel dispatch?

最も直接的な手段として試みた。ROCm の標準プロファイラである rocprofv3 で ollama serve を wrap し、 GPU kernel の dispatch 記録を取ることが目標だった。

This was the most direct approach. The goal was to wrap ollama serve with ROCm's standard profiler rocprofv3 to capture a record of GPU kernel dispatches.

何が起きたか：What happened: rocprofv3 の存在が Ollama の bootstrap discovery を壊し、GPU ランタイムの初期化に失敗する。 その結果、Ollama は GPU を認識できず CPU fallback に落ちた。 つまり、「GPU kernel を記録する」前に「GPU 自体が動かない」状態になった。 The presence of rocprofv3 broke Ollama's bootstrap discovery, causing GPU runtime initialization to fail. Ollama fell back to CPU. In other words, before any kernel could be recorded, the GPU itself was no longer being used.

ここから分かること： このデータは observer 汚染として扱う。CPU fallback での実行は RX9070XT の経路観測として意味をなさない。
→ 次の問い： wrap ではなく attach 方式なら経路を壊さずに記録できるか？

What this tells us: Any data from this run is treated as observer-contaminated. CPU-fallback execution is not a valid observation of the RX9070XT path.
→ Next question: Could attach mode avoid disrupting the runtime path?

手段② — rocprofv3 --attach（結果：ptrace ブロック） Approach 2 — rocprofv3 --attach (Result: ptrace blocked)

観測ポイント： 既に動いている Ollama プロセスに後からアタッチする方式なら、bootstrap を壊さずに記録できるか？

Observation target: If attaching to an already-running Ollama process, can we record kernels without breaking bootstrap?

wrap が bootstrap を壊すなら、Ollama を先に起動してから後でアタッチする方式を試みた。 これにより、bootstrap discovery は正常に完了させつつ、実行中の kernel を記録できる可能性があった。

Since wrap breaks bootstrap, the next attempt was to start Ollama first, then attach afterward. This would allow bootstrap discovery to complete normally before the profiler touches the process.

何が起きたか：What happened: ホストの ptrace_scope = 1 設定により、別プロセスへの ptrace アタッチが拒否された。 rocprofv3 --attach はこの制限に依存しており、変更なしには使用できない。 ptrace_scope を変更することはシステムセキュリティ設定の変更を意味し、調査環境の前提条件を変えてしまう。 The host's ptrace_scope = 1 setting rejected ptrace attachment to other processes. rocprofv3 --attach depends on this capability and cannot be used without changing this setting. Changing ptrace_scope means modifying system security settings, which alters the investigation environment's baseline.

ここから分かること： attach 方式は環境依存。ptrace_scope を下げれば動く可能性はあるが、それは「標準環境での観測」ではなくなる。
→ 次の問い： rocprofv3 以外の環境変数ベースの方法はあるか？

What this tells us: Attach mode is environment-dependent. Lowering ptrace_scope might work, but that would no longer be "observation in standard environment."
→ Next question: Are there environment-variable-based approaches that don't require ptrace?

手段③ — ROCBLAS_LAYER=9（結果：粒度不足） Approach 3 — ROCBLAS_LAYER=9 (Result: insufficient granularity)

観測ポイント： BLAS 層に絞って、rocBLAS の呼び出しを記録できるか？

Observation target: Can we at least log rocBLAS calls by targeting the BLAS layer?

rocprofv3 が使えない状況で、BLAS 層限定の logging を試みた。 ROCBLAS_LAYER=9 は rocBLAS の API 呼び出しをログに出す環境変数で、 runtime を壊さず使える可能性があった。

With rocprofv3 unavailable, we attempted BLAS-layer-only logging. ROCBLAS_LAYER=9 is an environment variable that logs rocBLAS API calls, and it potentially could be used without breaking the runtime.

何が起きたか：What happened: rocblas_create_handle の呼び出しは確認できた。しかし per-GEMM のログは出力されなかった。 つまり「rocBLAS が使われた（handle が作成された）」ことは分かるが、 「具体的にどの GEMM が、何回、どの kernel で実行されたか」は分からない。 The rocblas_create_handle call was confirmed. However, no per-GEMM logging was produced. This tells us "rocBLAS was used (handle was created)" but not "which specific GEMM ran, how many times, or which kernel handled it."

ここから分かること： BLAS 層にアクセスがあることは確認できた。しかし Q4_K の処理は MMVQ/MMQ 経路（custom kernel）が中心であり、 BLAS 経路に入らない限り ROCBLAS_LAYER からは何も見えない。 この手段は「BLAS fallback が起きているか」の確認には使えるが、custom kernel の観測には使えない。
→ 次の問い： これらすべてを除いた状態で、何か残るか？

What this tells us: BLAS layer access is confirmed. However, since Q4_K processing is dominated by MMVQ/MMQ (custom kernel paths), ROCBLAS_LAYER sees nothing unless the BLAS fallback path is taken. This approach can verify "is BLAS fallback occurring" but cannot observe custom kernels.
→ Next question: After eliminating all of these, what remains?

手段④ — Phase proxy（消去法の末に残った唯一の手段） Approach 4 — Phase Proxy (the last approach standing)

観測ポイント： runtime を一切壊さずに、「どの dispatch 圏で実行しているか」を近似的に読める手段があるか？

Observation target: Is there any approach that reads the approximate dispatch zone without touching the runtime at all?

手段①②③がすべて機能しない状況で、改めてソース側の dispatch 分岐を確認した。 MMVQ / MMQ の分岐は ne11（matrix の列数 ≈ prompt token 数）で決まる。 では、Ollama の response JSON に含まれる prompt_eval_count は ne11 の近似として使えないか？ これが phase proxy の発想。

With approaches 1–3 all blocked, we revisited the source dispatch logic. The MMVQ/MMQ branch is determined by ne11 (matrix columns ≈ prompt token count). Could prompt_eval_count in Ollama's response JSON serve as an approximation for ne11? This was the origin of the phase proxy approach.

何が起きたか（Phase F probe 結果）：What happened (Phase F probe results): prompt_eval_count = 3 → MMVQ 圏（ne11 ≤ 8 に対応）。 prompt_eval_count = 11 → MMQ-eligible 圏（8 < ne11 ≤ 256 に対応）。 runtime は壊れず、GPU 上で正常に実行された。 prompt_eval_count = 3 → MMVQ zone (maps to ne11 ≤ 8). prompt_eval_count = 11 → MMQ-eligible zone (maps to 8 < ne11 ≤ 256). The runtime was not disrupted; execution proceeded on GPU normally.

ここから分かること： phase proxy は「dispatch の直接確認」ではなく「dispatch 圏の近似推定」。 それでも、runtime を壊さずに dispatch 圏を区別できる現時点で唯一の手段であることが確認できた。

What this tells us: The phase proxy is not direct dispatch confirmation — it is an approximate inference of the dispatch zone. However, it has been confirmed as the only currently available means to distinguish dispatch zones without disrupting the runtime.

次の方法論ステップ — 量子化横断 dispatch map Next Method Step — Quantization-Agnostic Dispatch Map

direct observer がまだ blocked のままなので、次に広げるべきなのは workload を無秩序に増やすことではなく、 source 側で tensor type から dispatch family への地図を先に作ること。 ここでは、いまの Q4_K_M case study と、量子化方式をまたぐ一般 map を意図的に分ける。

With direct observers still blocked, the next expansion is not to add workloads indiscriminately, but to build a source-side map from tensor type to dispatch family first. We intentionally separate the current Q4_K_M case study from the broader cross-quantization map.

ここで重要なのは、file type と tensor type が同じではないこと。 `Q4_K_M` というラベルを持つモデルでも、実際の tensor 側では `Q4_K` だけでなく `Q5_K` / `Q6_K` / `Q8_0` が混ざりうる。 The key point is that file type and tensor type are not the same. Even a model labeled Q4_K_M can contain tensor-side mixes such as `Q4_K`, `Q5_K`, `Q6_K`, and `Q8_0`.

この一般 map を広げる最初の別枝としては IQ1_S を先に置く。 理由は、source 上で `MMVQ` / `MMQ` の両方に entry があり、しかも既存の RDNA4 fatbin には `bundle_0082` という `gfx1201` `mul_mat_q` anchor がすでに見えているから。 さらに practical path を分解すると、`ollama create -q IQ1_S` のような requantize route は狭いが、existing GGUF の direct import path は別経路で残っている。`Q5_K_M` はその次に回す。

The first branch to broaden this general map with is IQ1_S. The reason is that it already has source-side entry into both `MMVQ` and `MMQ`, and the current RDNA4 fatbin already exposes a `gfx1201` `mul_mat_q` anchor in `bundle_0082`. Practical-path analysis also split further: the `ollama create -q IQ1_S` requantize route is narrow, while direct import of an existing GGUF appears to remain a separate path. `Q5_K_M` comes after that.

follow-up では、この practical split を実機で確認した。small single-file `IQ1_S` GGUF (`Llama-3.2-1B-Instruct-UD-IQ1_S.gguf`) を `Modelfile` から direct import すると、 `ollama create -> show -> api/generate -> ollama ps` まで current host で成立し、resident case では `100% GPU` が見えた。 つまり current Ollama では、`IQ1_S` は「requantize helper は狭いが、existing GGUF direct import は通る」と書くのが正確。 A follow-up practical check confirmed this split on real hardware. A small single-file `IQ1_S` GGUF (`Llama-3.2-1B-Instruct-UD-IQ1_S.gguf`) successfully passed through `ollama create -> show -> api/generate -> ollama ps` on the current host, and the resident case showed `100% GPU`. So for current Ollama, the accurate statement is: `IQ1_S` has a narrow requantize helper path, but existing-GGUF direct import does work.
さらに bounded probe では、`IQ1_S` も `Q4_K` に近い二段構えで読めた。 binary 側では `bundle_0030` に `IQ1_S` MMVQ (`ggml_type19 1..8`)、`bundle_0082` に `IQ1_S` MMQ (`ggml_type19 8..128`) があり、 live prompt window でも `prompt_eval_count = 7` は `MMVQ-compatible`、`13` は `MMQ-eligible` と読むのが自然。 まだ exact dispatch proof ではないが、`IQ1_S` も source / binary / phase proxy の三層で narrowing できるところまで来た。 A bounded probe also suggests that `IQ1_S` can be read in a two-stage pattern similar to `Q4_K`. On the binary side, `bundle_0030` carries `IQ1_S` MMVQ (`ggml_type19 1..8`) while `bundle_0082` carries `IQ1_S` MMQ (`ggml_type19 8..128`). On the live side, `prompt_eval_count = 7` reads as `MMVQ-compatible`, while `13` reads as `MMQ-eligible`. This is still not exact dispatch proof, but it means `IQ1_S` can now be narrowed across source, binary, and phase-proxy layers.
