RX9070XT Optimization Method

RX9070XT への方法論移植の記録です。MI25 で確立した「観測してから最適化する」手順を rDNA4 固有の可視性制約に合わせて調整しています。

Records methodology transplantation to RX9070XT. The MI25 "observe first, then optimize" procedure is adapted to rDNA4's specific visibility constraints.

方法論の原則（MI25 由来・rDNA4 適用）Methodology Principles (MI25 Origin, rDNA4 Adaptation)

  • 観測クラスを先に確立する。観測できない状態で最適化を始めると、改善を証明できないだけでなく、退化も見逃す。
  • 多層証拠で経路を固める。ソース → 共有ライブラリ → バンドル分離 → 逆アセンブリ → ランタイム相関の順に証拠を積み上げ、各層の限界を明示する。
  • observer の影響を分離する。observer が runtime path を変えるなら、その observer は無効。observer によって path が変化したことが確認された場合は、そのデータを「observer 汚染」として記録し使用しない。
  • MI25 の signature を rDNA4 に転写しない。wavefront_size や kernel family が異なるため、MI25 の可視性をそのまま成功条件にしない。RDNA4 固有の見え方を別途確立する。
  • 単発 run で再現性を確定しない。複数 run で条件を明示した上での比較結果のみを観測事実として保持する。
  • Establish an observation class first. Starting optimization without observability means you cannot prove improvements — and cannot detect regressions either.
  • Layer evidence progressively. Build evidence across source → shared library → bundle isolation → disassembly → runtime correlation, making each layer's limits explicit.
  • Isolate observer effects. If an observer changes the runtime path, that observer is invalid for measurement. Data from an observer that changes path is recorded as "observer-contaminated" and not used.
  • Do not transfer MI25 signatures to rDNA4. wavefront_size and kernel families differ; MI25 visibility cannot be used as a success criterion for rDNA4. Establish rDNA4-specific observability separately.
  • Do not confirm reproducibility from a single run. Only comparison results with stated conditions across multiple runs are retained as observed facts.

調査フェーズの設計（C14 pilot 6 層）Investigation Phase Design (C14 Pilot 6 Layers)

Phase A
ソース footing 確立。一次コードソース・runtime truth・参照ツリーを固定。
Fix primary code source, runtime truth, and reference trees.
Phase B
ソースレベル経路追跡。呼び出しチェーンと dispatch 分岐候補を確定。実機 runtime 確認も含む。
Source-level path trace. Fix call chain and dispatch branch candidates. Includes live runtime confirmation.
Phase C
共有ライブラリ依存追跡。ldd で依存チェーンと bundle root を確定。
Shared library dependency map. Fix dependency chain and bundle root via ldd.
Phase D
コードオブジェクト目録。libggml-hip.so fatbin と rocBLAS library 内の gfx1201 オブジェクト family を列挙。
Code object inventory. Enumerate gfx1201 object families in libggml-hip.so fatbin and rocBLAS library.
Phase E
逆アセンブリによる絞り込み。Q4_K 候補 bundle を isolate し、register footprint・命令パターンを確定。
Disassembly-assisted narrowing. Isolate Q4_K candidate bundles and fix register footprints and instruction patterns.
Phase F
ランタイム・逆アセンブリ相関。Phase B–E 証拠を統合し、live case との相関を取る。Phase proxy probe を追加。
Runtime–disassembly correlation. Integrate Phase B–E evidence and correlate against live case. Add phase proxy probe.

観測クラスから最適化へ移行する条件Prerequisites for Moving from Observation Class to Optimization

現在未達の条件：Currently unmet prerequisite: dispatch-safe observer（runtime path を壊さずに per-dispatch kernel attribution ができる手段）の確保。 dispatch-safe observer (a method that can attribute per-dispatch kernels without distorting the runtime path).
  • dispatch-safe observer が確保できた後、比較単位（anchor workload・anchor conditions）を固定します。
  • single-knob 比較：knob を一つ変えて eval_duration / prompt_eval_duration の変化を記録します。
  • 複数 run で再現性を確認し、改善量を条件付きで記録します（「条件 X のもとで Y ms 改善」の形式）。
  • 改善の帰属先（kernel family / BLAS vs custom）を、observer の限界を明示した上で記録します。
  • After a dispatch-safe observer is secured, fix the comparison unit (anchor workload + anchor conditions).
  • Single-knob comparisons: change one knob and record changes in eval_duration / prompt_eval_duration.
  • Confirm reproducibility across multiple runs; record improvement with stated conditions ("Y ms improvement under condition X").
  • Record attribution of improvements (kernel family / BLAS vs custom) with explicit observer limitations stated.
